# Implementation Plan: Versionamento de Embeddings

**Branch**: `010-versionamento-embeddings` | **Date**: 2026-07-21 | **Spec**: `specs/010-versionamento-embeddings/spec.md`

**Input**: Feature specification from `/specs/010-versionamento-embeddings/spec.md`

## Summary

Add embedding provenance and compatibility checks so vectors generated by different providers, models, dimensions or revisions are never mixed silently. Persist effective embedding configuration in `embedding_batches`, attach chunks to batches with normalized content hashes, filter vector search to the active compatible configuration, expose pending/reindex state, and replace automatic vector-dimension mutation with an explicit startup guard.

## Technical Context

**Language/Version**: Python 3.13

**Primary Dependencies**: FastAPI, SQLAlchemy async, Pydantic v2, FastMCP, PostgreSQL + pgvector

**Storage**: PostgreSQL with pgvector; add `embedding_batches` and chunk provenance columns.

**Testing**: `.venv/bin/python -m pytest`

**Target Platform**: Linux web service with MCP server.

**Project Type**: Backend API + MCP search/ingestion integration.

**Performance Goals**: Preserve typical search under 500ms by using indexed batch/status filters before vector ordering; reindex operations should be bounded by explicit limits.

**Constraints**: Keep routes thin; business logic in `backend/app/services`; repository query changes in `backend/app/repositories`; update `doc/API.md` for public behavior; do not add a background worker in MVP.

**Scale/Scope**: Existing ingestion, source update, search, answer and MCP search/ingest paths. Optional operational endpoints for active config, pending chunks and bounded reindex.

## Implementation Details

### Settings and Active Config

- Add `embedding_version: str` to `backend/app/core/settings.py`.
- Derive an `EmbeddingConfigIdentity` from:
  - `llm_provider`
  - `embedding_model`
  - `vector_dim`
  - `embedding_version`
- Build a stable SHA-256 `config_hash` from normalized identity fields.
- Keep provider/model/dimension/version values as explicit columns for querying/debugging.

### Database and Init

- Add `EmbeddingBatch` SQLAlchemy model to `backend/app/db/models.py`.
- Add columns to `KnowledgeChunk`:
  - `embedding_batch_id`
  - `embedding_content_hash`
  - `embedding_status`
  - `embedded_at`
  - `embedding_error`
- Add idempotent schema additions in `backend/app/db/init.py`.
- Replace the current automatic `ALTER TABLE knowledge_chunks ALTER COLUMN embedding TYPE vector(768)` block with a dimension assertion:
  - read pgvector typmod for `knowledge_chunks.embedding`;
  - compare with `settings.vector_dim`;
  - raise `EmbeddingConfigurationError` or startup-specific error on mismatch.
- Existing rows:
  - non-null embedding + no batch -> `unversioned`;
  - null embedding -> `pending`.

### Ingestion

- In `ingest_text_source()`, create or start an `EmbeddingBatch` for the active identity before calling the provider.
- Compute `embedding_content_hash` for each normalized chunk content.
- Call embedding provider only for chunks without reusable compatible embedding, if reuse is implemented in MVP.
- Persist chunks with `embedding_status='embedded'`, `embedding_batch_id`, `embedding_content_hash` and `embedded_at`.
- On provider failure, avoid committing partial successful-looking version state.

### Source Update and Reindex

- Metadata-only source updates remain embedding-neutral.
- Content updates regenerate chunks and create/attach a fresh compatible batch.
- Add a pending detector that marks or lists chunks where:
  - batch missing;
  - status not `embedded`;
  - batch identity differs from active identity;
  - content hash differs from current normalized chunk content.
- Add bounded reindex service:
  - dry-run mode lists candidates;
  - execution mode embeds pending chunks for active identity;
  - idempotently skips/reuses chunks already embedded for the same hash/config.

### Search and Answer

- Extend `search_similar_chunks()` to join/filter by compatible `EmbeddingBatch`.
- Require `KnowledgeChunk.embedding_status == 'embedded'`.
- Apply compatibility filters before ordering by vector distance and before candidate limit.
- Keep existing category/tag/project filters.
- Keep `search_text_chunks()` available for text search; optionally annotate returned chunk read model with embedding status.
- Ensure hybrid fusion never labels an incompatible text-only result as `vector`.

### API/MCP

- Add read models for optional embedding status/config if exposed.
- Add optional endpoints:
  - `GET /knowledge/embeddings/config`
  - `GET /knowledge/embeddings/pending`
  - `POST /knowledge/embeddings/reindex`
- MCP search uses backend semantics automatically.
- Add MCP status/reindex tools only if REST endpoints are included; otherwise keep MCP unchanged except response model fields if public.

### Documentation

- Update `doc/API.md` for:
  - embedding compatibility semantics in search/answer;
  - operational endpoints if implemented;
  - migration/reindex notes for model/dimension changes.
- Keep `specs/010-versionamento-embeddings/contracts/embedding-versioning.md` aligned with final public shape.

## Data Model / API Implications

- Database gains `embedding_batches`.
- `knowledge_chunks` gains provenance/status columns.
- Search vector query gains batch/status compatibility predicates.
- Ingestion/source-update writes batch metadata.
- API may expose embedding state on chunk/detail responses and operational endpoints.
- Startup/init no longer auto-mutates vector dimension.

## Test Strategy

- Unit tests for config identity and hash stability.
- Init tests for vector dimension match/mismatch.
- Service tests for ingestion creating batch metadata and chunk hashes.
- Search tests proving incompatible/unversioned chunks are excluded from vector search.
- Hybrid tests proving text-only results do not claim vector match.
- Reindex tests for pending detection, config changes, idempotent hash reuse and provider failure.
- API integration tests for optional config/pending/reindex endpoints.
- MCP tests only if public response/tool contract changes.

## Risk Notes

- Existing unversioned data can disappear from vector results until reindexed. This is intentional but must be documented.
- Full-text fallback may surprise users if results continue appearing after model changes; `match_reasons` should make this explicit.
- Reusing embeddings by hash requires careful transaction handling to avoid copying stale/incompatible vectors.
- Dimension migration is operationally heavier than current init behavior; error messages must be concrete.
- Adding batch joins can slow vector search without indexes.
